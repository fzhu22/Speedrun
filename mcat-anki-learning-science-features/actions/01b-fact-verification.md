## ✅ GAP-2 / GAP-3 RESOLVED (Step-3 operator critique, primary-source-cited) — the mobile review-tier WRITE path
The biggest latent risk (skeptic + red-teamer both flagged it; A9-S1/A4-S3 quietly depended on it) is now answered against source. **Verdict: review-tier card-JS CANNOT reliably persist captured state into the synced collection cross-client without a desktop reconcile.** Concrete mechanisms:
1. **The only synced write channel from the review flow is `customData`, written AT ANSWER TIME by the custom-scheduling transform** (Anki unpacks `states.current.customData`, the transform mutates `.again/.hard/.good/.easy`, re-packs on grade) — ≤100 B, persists only when a grade button is pressed (`ts/reviewer/answering.ts`).
2. **The value captured in the card-template webview (sort result, confidence) is NOT an input to the scheduling transform** (its inputs are `states` + `{deck_name, seed, decay, desired_retention}`); bridging relies on a fragile shared JS global, and on AnkiDroid the customData-populate path has been broken (`states.current.customData` must be set before answering — AnkiDroid #14865; proto: "If it's not set when answering a card, the existing custom data will not be updated").
3. **AnkiDroid escape = the `AnkiDroidJS` API** — a FIXED menu (`ankiSetCardDue`/`ankiBuryCard`/`ankiSuspendCard`/`ankiMarkCard`/`ankiToggleFlag` + get/set **tags** per #17795) that DOES sync; requires `new AnkiDroidJS({version,developer})` + user-enabled JS API; some calls open a dialog. The community-blessed synced store is **tags** (no size cap; tag-list bloat the cost). No arbitrary customData write.
4. **AnkiMobile (iOS): NO supported path to write card data from the review screen** ("does not provide an API to modify card data directly… completely out of the question" — AnkiWeb forums). Only `sessionStorage` (front→back, one view) / `localStorage` (device-local, **not synced**, resets on app update). Nothing reaches the synced collection.
→ **Design rule for the revision:** desktop-prepare reconcile is the AUTHORITATIVE write path for all review-tier-captured state; mobile customData-at-answer is a desktop/Android-only, version-contingent optimization (test #14865 behavior on target versions, never assume); **iOS = capture-local / practice-only** for any gated feature (A3-S4 unlock, A7 confidence, A6-S3 remediation state). GAP-1 (CARS license), GAP-4 (estimator formula), GAP-5 (DR ramp policy) remain open.
**Also corrected (operator, primary-source):** (a) the **review tier runs on STOCK AnkiDroid/AnkiMobile with no fork build / no add-on** (AI-Hints precedent) — building AnkiDroid from source is NOT needed for the review tier; (b) **AnkiConnect is an HTTP server inside a running Qt GUI**, not an offline/cron service → "prepare tier" = a local single-collection script (headless = xvfb), not a multi-tenant backend; (c) **a schema column (native per-card-DR write) breaks stock-client + AnkiWeb sync** → forces own-sync-server + own-forked-clients; to keep stock clients the native write must target `customData`, not a new field (product-defining fork, not an A8 sub-bullet); (d) **note-type FIELD/template edits = schema-modifying = forced full sync** (iterate on a scratch profile; freeze fields early); (e) Anki is **single-writer (`with_col`)** → third-party writes while a device has unsynced changes trigger "full sync required / lose-changes" → prepare must be idempotent, resumable, paginated, backup-first, clients-quiesced; (f) **APKG round-trip does NOT cleanly preserve scheduling/customData and can duplicate on GUID collision** → demo transport, not a persistence/convergence story.
